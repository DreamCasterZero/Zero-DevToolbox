# 编程行为准则

以下准则适用于本项目中所有的代码编写、审查和重构工作，旨在减少常见的编码失误。源自 [Andrej Karpathy 对 LLM 编码陷阱的观察](https://x.com/karpathy/status/2015883857489522876)。

**权衡说明：** 这些准则偏向"谨慎优先于速度"。对于琐碎的小任务，可灵活判断。

## 1. 先思考，再编码

**不要臆测，不要掩盖困惑，主动暴露权衡。**

在动手实现之前：

- 明确陈述你的假设。如果不确定，就提问。
- 如果存在多种理解，把它们都列出来，不要默默选一个。
- 如果有更简单的方案，说出来。该反对时就反对。
- 如果有不清楚的地方，停下来。指出困惑点，提问。

## 2. 简单优先

**用解决问题的最少代码，不写任何投机性的东西。**

- 不实现超出需求范围的功能。
- 不为只用一次的代码做抽象。
- 不添加没有被要求的"灵活性"或"可配置性"。
- 不为不可能发生的情况写错误处理。
- 如果写了 200 行而其实 50 行就够，那就重写。

自问："资深工程师会觉得这写得过度复杂吗？" 如果会，就简化。

## 3. 外科手术式改动

**只动你必须动的地方。只清理你自己制造的混乱。**

修改现有代码时：

- 不要"顺手改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 沿用现有风格，即使你自己更倾向别的写法。
- 如果发现无关的死代码，提一句即可，不要删除。

当你的改动产生了"孤儿"时：

- 移除因你的改动而不再被使用的 import/变量/函数。
- 不要删除原本就存在的死代码，除非被明确要求。

检验标准：每一行被改动的代码，都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准，循环验证直到达成。**

把任务转化为可验证的目标：

- "加个校验" → "为非法输入写测试，然后让测试通过"
- "修复这个 bug" → "写一个能复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，先给出简要计划：

```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强的成功标准能让你独立循环推进；弱的标准（如"让它能用"）会导致反复来回确认。
