# 双语 README 生成

为当前项目生成两个互相链接的 README 文件：`README.md`(英文，GitHub 默认显示)和 `README_zh.md`(简体中文)。

## 执行步骤

### 1. 调研项目

先充分了解项目，不要凭空编造内容：

- 读取项目根目录结构（`ls`、`view` 目录）
- 读取关键文件：主入口脚本、`requirements.txt`/`pyproject.toml`/`package.json`、已有的配置文件
- 阅读核心代码，理解项目**实际做什么**、用了什么算法/技术、怎么运行
- 如果有论文链接、已有文档、注释，一并参考
- 识别技术栈、依赖、Python/语言版本

**重要：只写代码里真实存在的东西。** 不确定的功能不要写；安装步骤、运行命令必须基于项目实际情况，不要套用通用模板里的假命令。

### 2. 语言切换链接

每个文件**第一行**放语言导航（这是 GitHub 上实现"语言切换"的标准做法——本质是两个文件互相跳转）：

`README.md` 顶部：

```markdown
[English](README.md) | [简体中文](README_zh.md)
```

`README_zh.md` 顶部：

```markdown
[English](README.md) | [简体中文](README_zh.md)
```

### 3. 结构（两个文件结构一致，只是语言不同）

按以下顺序，两者兼顾"面试官/同行了解技术亮点"和"使用者快速上手"：

1. **语言切换链接**（第一行）
2. **项目标题 + 一句话简介**
3. **Badges**：根据项目实际情况选择，常用的有：
   - 语言/版本：`![Python](https://img.shields.io/badge/Python-3.11-blue)`
   - 许可证：`![License](https://img.shields.io/badge/License-MIT-green)`（先确认仓库有无 LICENSE 文件，没有就不加或提示用户补）
   - 其他贴合项目的（如算法领域、论文状态）
   - 不要堆砌无意义的 badge，3-5 个相关的即可
4. **目录（Table of Contents）**：用锚点链接，方便长文档跳转
5. **简介 / 背景**（Introduction）：项目解决什么问题、核心思路、技术亮点。**这部分是给同行和面试官看的重点**，把算法/方法的价值讲清楚（如果是研究型项目，说明用了什么方法、有什么效果/指标）
6. **特性（Features）**：要点列表，突出技术亮点
7. **项目结构（Project Structure）**：目录树 + 简短说明（用代码块画目录树，但不要画复杂架构图）
8. **安装（Installation）**：基于项目真实依赖写，包含环境要求、依赖安装命令
9. **使用方法（Usage）**：怎么运行，给出真实可执行的命令和最小示例
10. **结果 / 示例（Results / Examples）**：如果项目有实验结果、性能指标、可视化图，在这里展示（研究型项目尤其重要）
11. **技术栈（Tech Stack）**：用到的主要库/工具
12. **许可证（License）**：与 LICENSE 文件一致
13. **引用 / 致谢（Citation / Acknowledgements）**：如果有关联论文，给出引用格式；如有参考的开源项目，致谢

不是每个项目都需要全部章节，按项目实际情况裁剪。研究型/算法项目重点放在简介、特性、结果；工具型项目重点放在安装、用法。

### 4. 写作要求

- 英文版用地道、简洁的技术英语，不要中式英语
- 中文版用专业、清晰的中文，不要把英文生硬直译
- 两个文件内容对应一致，但各自符合该语言的表达习惯
- 代码块标注语言（`python、`bash 等）
- 标题层级合理，目录锚点和实际标题对得上

### 5. 输出

- 在项目根目录创建 `README.md` 和 `README_zh.md`
- 创建后简要说明：写了哪些章节、哪些信息是基于代码推断的（需要用户确认）、哪些占位需要用户补充（如 LICENSE、论文链接、实验数据）

## 注意事项

- 不臆造功能、不写代码里没有的东西
- 不确定的地方用占位符并明确标注，让用户补充，而不是编造
- badge 链接确保格式正确可访问
- 如果项目已有 README，先读它，在其基础上改进而不是完全推翻
